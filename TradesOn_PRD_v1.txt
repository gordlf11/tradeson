1. Document Overview
This Product Requirements Document (PRD) defines the full functional scope of the TradesOn Platform — a marketplace connecting homeowners, property managers, and realtors with licensed and non-licensed trade service providers. It serves as the primary reference for engineering, design, and QA teams to build toward a Technical Design Document (TDD).

This PRD is structured to support a phased delivery model:
	•	Phase 1 — Web Application (MVP): A fully responsive web app that delivers core functionality without requiring native mobile development. Prioritizes speed to market and enables early user validation.
	•	Phase 2 — Native Mobile App (Beta): Expands into a native iOS/Android experience with capabilities not feasible in the browser, such as live GPS tracking, push notifications, and offline mode.

1.1 Product Vision
TradesOn is an AI-powered home services marketplace that eliminates the friction of finding, vetting, and hiring trade professionals. The platform serves multiple user roles — homeowners, property managers, realtors, and tradespeople — with role-specific workflows, AI-assisted job intake, automated compliance verification, and integrated payments.
1.2 User Roles
The platform serves five distinct user types, each with a tailored experience:
	•	Homeowner — Creates and manages home repair/service jobs; hires tradespeople.
	•	Property Manager — Manages multiple properties; assigns and oversees jobs at scale.
	•	Realtor — Coordinates service work for buyer/seller transactions; may be managed under a Brokerage account.
	•	Licensed Tradesperson — Verified professional with state credentials; browses and bids on jobs.
	•	Non-Licensed Tradesperson — General service provider; undergoes identity and document verification.
	•	Platform Admin — Internal staff managing compliance, user governance, and platform health.

2. Functional Requirements
(MVP)Function 1 — Authentication  & Account Management
Handles user registration, login, and session management. All roles access the platform through a shared authentication gateway that routes users to their role-specific experience post-login.
Workflow
	•	User lands on the Login Page and enters credentials (email + password).
	•	If the user does not have an account, they are directed to the Account Creation flow.
	•	Upon successful registration, the user proceeds to Role Selection (Screen 3) which routes them to the appropriate onboarding flow.
	•	Returning users are authenticated and directed to their role-specific dashboard.

Web App (MVP): Use JWT-based session tokens stored in httpOnly cookies. OAuth (Google) sign-in can be added in a follow-on sprint.
Native App (Beta): Add biometric authentication (Face ID / Touch ID) at the native layer. Deep link handling for magic link emails.

Function 2 — Onboarding (POC) & Identity Verification (API - MVP)
Each user role has a distinct onboarding flow that collects role-appropriate information, payment details, and identity documentation. Tradesperson onboarding includes AI-assisted document verification via a Compliance Agent.
Workflow — Tradesperson Onboarding (Licensed & Non-Licensed)
	•	The provider begins registration and selects their trade role.
	•	Completes a Trades Profile: personal details, trade categories, and service radius (drawn on an interactive map).
	•	Uploads required documents: government-issued ID, professional license (licensed only), and proof of insurance.
	•	Compliance Agent activates and begins automated verification:
	•	ID Verification API validates identity.
	•	License Verification checks professional credentials against state databases.
	•	Insurance Validation confirms coverage type, limits, and policy expiration.
	•	If any document fails verification, the provider is notified and prompted to resubmit corrected documentation.
	•	Upon successful verification, a Verified Badge is assigned and access to the Job Board is granted.

Web App (MVP): Service radius drawn via Google Maps JavaScript API embedded in the browser. Document upload uses standard file input (accept='.pdf,.jpg,.png'). Async verification status shown via polling.
Native App (Beta): Service radius drawn via native map SDK gestures. Camera capture for document upload. Push notification when verification is complete.

Workflow — Customer Roles (Homeowner, Property Manager, Realtor)
	•	User selects their role on the User Type Selection screen.
	•	Completes role-specific profile form.
	•	Sets up a payment method via the integrated payment processor (Stripe).
	•	Completes identity verification (Stripe Identity or equivalent).
	•	Account activated and user is redirected to their role-specific dashboard.

Workflow

Web App (MVP): All brokerage management surfaces are web-first. Role-based access control (RBAC) enforced at API layer. Billing via Stripe Connect or billing portal.

Function 4 — Admin Portal (MVP)
An internal-facing control center for platform administrators to review compliance submissions, monitor account health, resolve disputes, and track platform-level metrics.
4.1 Governance & Compliance Review
Admins review verification submissions from tradespeople and make approval decisions.
	•	Review submitted documents: government ID, professional license, insurance certificate.
	•	Cross-reference license numbers against state databases.
	•	Validate insurance policy coverage, limits, and expiration dates.
	•	Add internal admin notes to the review record.
	•	Make a decision:
	•	Approve → Account is activated and Verified Badge is assigned.
	•	Reject → User is notified with a reason; account remains inactive.
	•	Request More Documents → User is notified with specific instructions.

4.2 Account Monitoring & Flag Detection
The system automatically surfaces accounts that require admin attention based on behavioral and status signals:
	•	Open payment disputes or failed chargebacks.
	•	Consistently poor service reviews (below threshold).
	•	Expired insurance certificates.
	•	Suspicious activity patterns (e.g., multiple failed logins, unusual job volume).

4.3 Admin Resolutions
Admins have a structured set of resolution actions available for flagged accounts:
	•	Send a formal warning with required acknowledgment.
	•	Request a written explanation from the user.
	•	Issue a temporary account suspension with a defined reinstatement date.
	•	Issue a permanent account deactivation with documented reason.

4.4 Audit Log
All admin actions are immutably logged to support accountability, audits, and dispute resolution. Each log entry captures: action type, acting admin user, target account, timestamp, and documented reason.

4.5 Platform Metrics Dashboard
Aggregated platform health metrics available in real time to admins. Core metrics include:
	•	Total Registered Users (by role)
	•	Monthly Active Users (MAU) — overall and by role
	•	Active Jobs — open, in-progress, and completed
	•	Gross Revenue and Net Revenue (after platform fee)
	•	Operational Expenses

4.6 Funnel & Behavioral Metrics
Funnel and behavioral tracking to support growth decisions:
	•	Customer acquisition funnel: visit → sign-up → onboarded → first job created.
	•	Tradesperson acquisition funnel: sign-up → verified → first bid submitted → first job won.
	•	Supply/demand ratio monitoring by ZIP code and trade category.
	•	Activation rate tracking with automated email triggers for inactive users.
	•	Dynamic tradesperson recruitment pausing when supply exceeds demand in a region.

Function 6 — Job Creation (AI Intake) (POC)
The AI Intake system allows customers to describe a home service need via photo, voice, or text. An AI Intake Agent processes the input, classifies the job, scores urgency/severity, generates a preliminary cost estimate, and drafts a job summary for user confirmation.
Workflow
	•	User initiates a new job by selecting 'Create New Job.'
	•	User submits input via one or more channels: photo upload, voice recording, or free-text description.
	•	Create a uniform form structure:Room
	•	Type of Trade needed: Plumbing, electrical, etc.
	•	Severity
	•	Description Box
	•	User selects the trade classification from a structured category list.
	•	Enforce required inputs to validate that all required inputs are present.
	•	MVP - NLP and computer vision models analyze the input to extract issue details.
	•	MVP - Severity and urgency scoring is calculated; additional inputs are requested if confidence is insufficient.
	•	MVP - A preliminary cost range is generated based on trade type, severity, and regional pricing data.
	•	MVP - The AI drafts a structured job summary (issue description, trade category, severity, cost range).
	•	The user reviews the summary, edits if needed, and approves.
	•	Approved jobs are routed to the Job Board.

Web App (MVP): Photo upload via standard file input. Voice input via browser MediaRecorder API (limited cross-browser support — fallback to text). AI processing status shown via animated loading state. Inline editing of AI-generated summary fields.
Native App (Beta): Native camera and microphone access with real-time capture. AI processing in background with push notification on completion. Richer voice-to-text with on-device processing.

MVP - AI Agent Validation Logic
	•	Flag jobs where photo quality is insufficient (blurry, dark, or irrelevant subject) and prompt for re-upload.
	•	Flag jobs where the described issue does not align with the selected trade category and surface a correction prompt.
	•	Request additional detail if severity cannot be confidently scored from the initial input.

Function 7 — Job Board & Quoting (P)C)
Approved jobs are posted to a filterable Job Board where verified tradespeople can discover, review, and submit quotes. Customers review all submitted quotes and select the best fit. Jobs expire if no quote is accepted within a defined window.
Workflow
	•	Approved job is entered into the Job Board routing system with a status of 'Open.'
	•	Tradespeople filter the board by distance (5 / 10 / 25 miles) and sort by newest, closest, or soonest-expiring.
	•	A tradesperson reviews the full job summary and submits a quote (price + message).
	•	Multiple tradespeople can submit quotes simultaneously (unlimited quote system).
	•	Customer receives notification of new quotes and reviews them in a comparison view.
	•	Customer selects one quote or allows the job to expire.
	•	Quote Accepted → Job assigned; move to Scheduling.
	•	Job Expired or All Quotes Declined → Customer must resubmit or cancel the job.

Job Expiration Rules
Open Design Decision: Expiration window to be defined (recommended starting point: 72 hours from posting). Expired jobs are archived and not re-posted automatically — customer must take explicit action.

Web App (MVP): Job Board rendered as a paginated list with filter sidebar. Quote submission via modal form. Customer quote comparison via side-by-side card layout.
Native App (Beta): Job Board supports swipe gestures for quick actions (save, skip). Native notifications for new quotes. Map view toggle for geospatial job browsing.

Function 8 — Scheduling (POC)
Once a quote is accepted, the platform facilitates scheduling coordination between the customer and the tradesperson. Pre-authorization of payment is captured before the appointment is confirmed.
Workflow
	•	Job acceptance triggers the scheduling flow; tradesperson's available time slots are surfaced to the customer.
	•	Customer selects 2–3 preferred time slots.
	•	Customer submits their time slot preferences.
	•	Tradesperson reviews options and selects the time that works best.
	•	Appointment is confirmed and both parties receive notification.
	•	Platform requests a payment pre-authorization (hold) for the quoted amount via Stripe.
	•	Customer authorizes the payment hold; funds are reserved but not captured.

Web App (MVP): Date/time picker built with a standard UI library (e.g., React Datepicker). Availability shown as a calendar grid. Confirmation sent via email. Payment pre-auth via Stripe PaymentIntents.
Native App (Beta): Native calendar integration for availability import. In-app scheduling confirmation with one-tap accept. Push notification for appointment confirmation.

Function 9 — Job Execution (POC)
Covers the end-to-end execution lifecycle: tradesperson traveling to the site, performing the work, handling scope changes, and completing the job for customer approval.
Workflow
	•	Job status updates to 'En Route' and live location tracking activates.
	•	Real-time map and ETA are displayed to the customer.
	•	On arrival, tradesperson marks 'Onsite' — live tracking ends and work begins.
	•	Tradesperson conducts an initial inspection to assess actual scope vs. quoted scope.
	•	If additional work is required:
	•	Tradesperson submits an additional work scope with description, photos, and revised pricing.
	•	Customer receives a notification and can accept or decline.
	•	If declined, work proceeds under the original quote (no job cancellation).
	•	If accepted, the total quoted amount is updated and logged.
	•	Work proceeds ('In Progress' status).
	•	Tradesperson marks work as 'Complete' and submits completion documentation.
	•	Customer reviews the work and approves; payment capture is triggered.

Web App (MVP): Live tracking uses Google Maps JavaScript API with 5-second polling refresh. Scope change approval is a web form modal. All quote adjustments logged in a visible history table.
Native App (Beta): Live tracking via native Maps SDK with WebSocket real-time updates. In-app approval flow for scope changes with one-tap accept/decline. Photo capture from device camera for completion documentation.

Scope Change Rules
	•	Additional work requests are non-blocking — the job continues under the original scope if declined.
	•	All quote adjustments (additions and declines) are logged chronologically in the Quote History Log.
	•	Customers cannot cancel a job mid-execution due to a scope change request.

Function 10 — Job Completion & Invoicing (POC)
After the customer approves the completed work, the platform generates a detailed invoice, processes payment, and closes the job record.
Workflow
	•	Tradesperson marks the job as 'Complete' in the platform.
	•	Tradesperson uploads completion photos and adds notes summarizing work performed.
	•	System calculates invoice line items: labor, materials, and any approved scope additions.
	•	A PDF invoice is automatically generated and delivered to the customer via email and SMS with a payment link.
	•	Customer reviews and approves the invoice in-app.
	•	Payment is captured from the pre-authorized hold (Stripe). Platform fee is applied.
	•	Job status updates to 'Closed.' Both parties can leave a review.

Web App (MVP): PDF invoice generated server-side (e.g., using a PDF library). Payment capture via Stripe PaymentIntents confirm call. Invoice delivered via Twilio SMS + SendGrid email.

Function 11 — Payments & Payouts (MVP)
Manages all payment flows on the platform: customer payment methods, tradesperson payout configuration, pre-authorization, capture, and job cancellation handling.
Platform Payment Model
	•	Customers are charged via pre-authorized Stripe PaymentIntents — funds are held at quote acceptance and captured upon job approval.
	•	Tradespeople receive payouts via Stripe Connect (or equivalent) on a configurable schedule (daily, weekly).
	•	A platform fee is applied to each completed transaction (percentage TBD by business).
	•	Disputed charges are surfaced in the Admin Portal for resolution.

Web App (MVP): Stripe.js Elements for PCI-compliant card input. Stripe Connect for tradesperson payout onboarding. Stripe Customer Portal for self-service billing management.
Native App (Beta): Stripe native iOS/Android SDK. Apple Pay and Google Pay support. Biometric-authorized payment confirmation.

Cancellation Policy
Job cancellation rules are designed to protect both parties. Open Design Decision: Specific fee structure TBD (e.g., free cancellation before tradesperson accepts; partial fee if cancelled after scheduling; tradesperson cancellation penalties). Cancellation can be initiated by either party and is handled within the platform.
