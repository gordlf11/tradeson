Foundations

Transport & Auth

  • All UI ↔ API calls use HTTPS / TLS 1.2+ (Cloud Run managed cert).
  • All API calls carry Authorization: Bearer <firebase_id_token>, an RS256 JWT (1h lifetime, auto-refreshed).
  • Admin routes require an additional custom claim: admin: true.
  • Real-time chat uses Firestore WebSocket channel via the Firebase JS SDK (onSnapshot).

Idempotency & Fallback

  • Mutating routes accept Idempotency-Key (client-generated UUID v4) for safe retry within 24h.
  • Onboarding writes are wrapped in PG transactions — rollback on partial failure.
  • API call failures during onboarding are non-blocking: UI navigates forward, localStorage stores breadcrumbs, self-heal runs on next login.
  • Pub/Sub deliveries are at-least-once with 7-day retention + exponential backoff. Consumers (Cloud Function fcm-fanout) are written to be idempotent.

Failure paths:
  • 503 from API (DB unreachable) → UI still navigates to /job-board (non-blocking pattern). User re-tries on next session.
  • Pub/Sub publish failure → logged + swallowed. The job exists in PG; the missing push notification can be backfilled via a delta job (not currently scheduled — acceptable at current scale).
  • FCM token-not-registered → fan-out function cleans up the stale token in Firestore.

Two different real-time channels: chat uses Firestore onSnapshot (WebSocket-backed); status changes go through REST + Pub/Sub → FCM.

5 — Payout on Completion

The hardest path. TP marks done → 3-hour HO confirmation window → Stripe capture + Connect transfer minus 10% platform fee. Cloud Scheduler auto-releases stalled holds.

sequenceDiagram autonumber participant TP as TP Browser participant HO as HO Browser participant API as Cloud Run API participant PG as Postgres participant ST as Stripe participant CS as Cloud Scheduler participant PS as Pub/Sub participant FN as fcm-fanout TP->>API: PATCH jobs status pending_confirmation API->>PG: UPDATE jobs status, auto_release_at now plus 3h API-)PS: publish job.status_changed pending_confirmation PS->>FN: deliver event FN->>HO: push "Confirm and release payment" alt HO confirms within 3h HO->>API: POST jobs confirm-complete API->>ST: paymentIntents.capture pi_xxx ST-->>API: charge succeeded API->>ST: transfers.create amount times 0.90 to tp_stripe_acct ST-->>API: transfer created API->>PG: UPDATE jobs status completed, UPDATE payments status completed API-)PS: publish job.status_changed completed PS->>FN: deliver event FN->>TP: push "You've been paid!" else HO inactive after 3h CS->>API: POST internal release-expired-holds every 30 min API->>PG: SELECT jobs WHERE pending_confirmation AND expired loop each expired job API->>ST: capture and transfer API->>PG: UPDATE jobs and payments end end
▸ Command — confirm_complete_command 
POST /api/v1/jobs/job_01H.../confirm-complete

// 2. Inbound from Stripe later:
POST /api/v1/stripe/webhooks (signature-verified)
{
  "type": "transfer.created",
  "data": { "object": { "id": "tr_..." } }
}
Schema boundary: Money never lives in our DB. Authoritative balance state is in Stripe; we store only Stripe object IDs (pi_, tr_, acct_) + denormalized amounts for UX display.
Failure path: If Stripe capture fails (card declined post-auth), the API returns 402 to the HO confirm call; job stays in pending_confirmation; admin gets a flagged_account row. The auto-release cron will retry on its next pass (every 30 min).
Authorization on the cron: X-Internal-Secret header matches a Cloud Run env var; the route 401s otherwise. Cloud Scheduler is the only caller in production.
